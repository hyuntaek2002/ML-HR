# ML-HR 발표 자료(PPT) 구성 가이드

## 1. 서론 및 개요 (Introduction & Overview)
**목표**: "우리가 왜 이 프로젝트를 했고, 한 줄로 요약하면 무엇인가?"를 명확히 각인시킵니다.

- **슬라이드 1: 프로젝트 배경 및 문제 정의**
  - 하루 수만 건씩 쏟아지는 뉴스, 사람이 다 읽을 수 없다.
  - LLM 발전으로 요약은 쉬워졌지만, "어떤 모델이 가장 잘하는가?" "자동으로 계속 돌아가게 할 수는 없나?"에 대한 MLOps적 해결책 부재.
- **슬라이드 2: 핵심 목표 (한 줄 요약)**
  - "네이버 뉴스 8개 분야를 8시간 주기로 자동 수집 → 3개 AI 모델 요약 → 하이브리드 심사위원 채점 → MLOps 추적까지 수행하는 **자율 주행 한국어 뉴스 요약 파이프라인**"
- **슬라이드 3: 전체 아키텍처 다이어그램**
  - DataOps -> ModelOps -> Evaluation -> Serving으로 이어지는 흐름을 시각화.
  - (Github의 `arccode.mmd` 다이어그램 기반으로 깔끔하게 리디자인)

---

## 2. 데이터 및 모델 설명 (DataOps & ModelOps)
**목표**: 파이프라인의 전반부(수집과 추론)를 어떻게 안정적으로 자동화했는지 어필합니다.

- **슬라이드 1: DataOps (실시간 데이터 파이프라인)**
  - 네이버 API 연동 및 8개 분야(IT, 경제 등) 타겟팅.
  - 정규식을 활용한 HTML 태그 및 노이즈 클리닝 로직 소개.
  - Supabase(PostgreSQL) 기반의 클라우드 실시간 적재.
- **슬라이드 2: ModelOps (왜 멀티 모델인가?)**
  - KoBART(인코더-디코더), KoT5(Text-to-Text) 등 **서로 다른 아키텍처의 3개 모델 벤치마킹**.
  - **기술적 포인트**: 모델 연속 추론 시 발생하는 GPU OOM(메모리 부족)을 막기 위한 명시적 가비지 컬렉션(`del model`) 및 캐시 초기화 전략 어필.

---

## 3. 실험 및 평가 결과 (Evaluation & MLflow) - 🌟 핵심 차별점
**목표**: 이 프로젝트의 가장 큰 자랑거리인 "다각화된 채점 시스템"과 추적 인프라를 강조합니다.

- **슬라이드 1: 5중 하이브리드 AI 심사위원 도입 배경**
  - 기존의 ROUGE 지표는 생성형 요약 모델의 '의미적 정확도'를 담아내지 못함.
  - 이를 대체할 새로운 채점 체계의 필요성 대두.
- **슬라이드 2: 이종(Cross) LLM 평가 + 수학적 유사도 결합**
  - **정성 평가 (70%)**: GPT-4o-mini (OpenAI) + HyperCLOVA X (Naver) 교차 채점. -> *단일 모델의 편향성을 막기 위한 이종 LLM 앙상블 전략 강조*
  - **정량 평가 (30%)**: BERTScore 방식의 고차원 벡터 코사인 유사도 수학적 연산.
- **슬라이드 3: MLflow 연동 (실험 추적)**
  - 8시간마다 실행되는 파이프라인의 채점 결과를 MLflow Tracking 서버에 연속적으로 기록.
  - 모델별 점수 추이를 대시보드로 시각화하여 모니터링.

## 4. 배포 및 운영전략 (Deployment & Observability)
**목표**: "단순히 한 번 돌려본 코드가 아니라, 실제 서비스처럼 운영된다"는 점을 보여줍니다.

- **슬라이드 1: CI/CD 및 완전 자동화**
  - GitHub Actions를 활용한 8시간 주기(cron) 크론잡 설정.
  - 로컬 환경(`schedule` 라이브러리)과 클라우드 환경을 자동 분기하는 런타임 대응 로직.
- **슬라이드 2: API 서빙 및 대시보드 (Streamlit & FastAPI)**
  - Streamlit: 분야별 1위 모델(Champion) 현황 및 최근 뉴스 3사 요약 결과 직접 비교 피드.
  - FastAPI: 외부에서 1위 모델을 조회하거나 파이프라인을 수동 트리거할 수 있는 백엔드 API 제공.
- **슬라이드 3: 자율 랭킹 및 챔피언 교체 로직**
  - 분야별 최근 20건의 평균 점수를 계산하여 항상 '가장 폼이 좋은' 챔피언 모델을 자동 선정.

## 5. 결론 및 향후 개선 (Conclusion & Future Work)
**목표**: 성과를 정리하고, 한계점을 "우리가 몰라서 안 한 게 아니라, 우선순위상 향후 과제로 뺐다"는 뉘앙스로 방어합니다.

- **슬라이드 1: 프로젝트 성과 요약**
  - 수집-요약-평가-배포까지 완전 자동화된 End-to-End MLOps 인프라 구축 성공.
- **슬라이드 2: 한계점 및 개선 방향 (방어용 쉴드)**
  - **평가 지표의 학술적 검증 부재 및 대안**: LLM-as-Judge 채점의 주관성을 극복하기 위해, 인간 평가와 상관관계가 증명된 학술 지표(QAFactEval, G-Eval) 도입을 시도했으나, 한국어 전용 QA 파이프라인 모델 부재 및 API 기술적 제약으로 현재의 하이브리드 방식으로 선회.
  - **데이터 드리프트 관점**: 현재는 '출력 점수 하락' 기반의 모니터링을 수행 중. 향후에는 입력 데이터 텍스트 분포 변화(PSI 검정 등)를 감지하는 진짜 데이터 드리프트 시스템 구축 예정.
  - **테스트 코드(CI) 도입 필요성**: 안정적인 운영을 위해 Pytest 기반의 단위/통합 테스트 코드 도입 예정.
