## 1. 드리프트 감지 — 접근 방법

현재 코드의 문제는 **"점수가 낮다 = 드리프트"** 로 판단한다는 점입니다:

```python
# main.py L52-56 (현재 방식)
if top_score >= 65:
    status = "✅ 정상"
else:
    status = "⚠️ 성능 하락(드리프트 의심)"
```

이것은 **모델 성능 모니터링**이지, 입력 데이터의 분포가 바뀌었는지를 보는 **데이터 드리프트**가 아닙니다.

### 진짜 데이터 드리프트란?

```
예시: 평소에 "경제" 분야 뉴스가 500~1000자의 기사 위주였는데,
      어느 날부터 200자 미만의 속보성 단문 기사가 쏟아지기 시작함
      → 모델이 학습한 패턴과 입력 분포가 달라짐 → 요약 품질 하락
```

**감지해야 하는 것**: 입력 텍스트 자체의 통계적 분포 변화

### 실현 가능한 접근법 (코드 수정 없이 발표용으로 설명할 수 있는 수준)

| 방법 | 측정 대상 | 난이도 |
|------|----------|--------|
| **텍스트 길이 분포** | 과거 50건 vs 최근 10건의 `len(description)` 평균/분산 비교 | ⭐ 쉬움 |
| **어휘 다양성** | 과거 vs 현재의 고유 단어 수(Type-Token Ratio) 비교 | ⭐ 쉬움 |
| **임베딩 거리** | 과거 기사들의 평균 임베딩 벡터 vs 현재 기사 임베딩의 코사인 거리 | ⭐⭐ 중간 |
| **PSI(Population Stability Index)** | 위 특징들의 분포 이동을 통계적으로 검정 | ⭐⭐⭐ 어려움 |

참고로 LYT 버전의 `data_drift.py`가 바로 이 방향(Readability + Burstiness + Perplexity → Evidently PSI)이었는데, `textstat`의 Flesch Reading Ease가 **한국어에 맞지 않는 영어 전용 공식**이라는 문제가 있었습니다.

### 발표에서의 포지셔닝

> "현재는 모델 출력 점수 기반의 **성능 드리프트 감지**를 구현했으며, 입력 데이터 자체의 통계적 분포 변화를 감지하는 **데이터 드리프트**(예: Evidently AI PSI 검정)는 향후 과제로 남겨두었다"

이렇게 두 개념을 **구분해서 설명**하면, 오히려 개념을 정확히 이해하고 있다는 인상을 줄 수 있습니다.

## 2. 테스트 코드 — 상세 설명

### 테스트 코드란?

"내가 만든 함수가 의도대로 동작하는지"를 **자동으로 검증하는 코드**입니다. Python에서는 `pytest`를 표준적으로 사용합니다.

### 이 프로젝트에서 테스트할 수 있는 것들

**a) 단위 테스트 (Unit Test)** — 개별 함수가 올바르게 동작하는지

```python
# tests/test_preprocess.py (예시)
from src.preprocess import clean_text

def test_html_tag_removal():
    """HTML 태그가 정상적으로 제거되는지"""
    assert clean_text("<b>안녕</b>하세요") == "안녕하세요"

def test_email_removal():
    """이메일 주소가 정상적으로 제거되는지"""
    assert "test@naver.com" not in clean_text("문의는 test@naver.com으로")

def test_multiple_spaces():
    """다중 공백이 하나로 압축되는지"""
    assert clean_text("단어   사이   공백") == "단어 사이 공백"
```

**b) 채점 로직 테스트** — 가중 앙상블이 올바르게 계산되는지

```python
# tests/test_evaluate.py (예시)
def test_hybrid_score_calculation():
    """정성 70% + 수학 30% 가중치가 정확한지"""
    # s1=80, s2=70, s4=90 일 때
    avg_llm = (80 + 70) / 2  # = 75
    expected = (75 * 0.7) + (90 * 0.3)  # = 52.5 + 27 = 79.5
    assert expected == 79.5

def test_score_clipping():
    """100점 초과 / 0점 미만 클리핑이 동작하는지"""
    assert min(100.0, 120.0) == 100.0
    assert max(0.0, -5.0) == 0.0
```

**c) 통합 테스트** — 파이프라인 흐름이 끊기지 않는지

```python
# tests/test_pipeline.py (예시)
def test_collect_returns_valid_data():
    """수집된 뉴스에 title과 description이 존재하는지"""
    # mock API 응답으로 검증

def test_model_generates_nonempty_summary():
    """모델이 빈 문자열이 아닌 요약을 반환하는지"""
    # 짧은 테스트 텍스트로 검증
```

### 왜 중요한가?

| 상황 | 테스트 없이 | 테스트 있으면 |
|------|-----------|-------------|
| `preprocess.py` 수정 후 | 파이프라인 전체를 돌려봐야 확인 가능 | `pytest tests/` 3초면 확인 |
| GitHub Actions에서 | 코드가 깨져도 배포됨 | **CI 단계에서 차단** |
| 발표 질의 시 | "테스트는 안 했습니다" | "pytest 기반 단위 테스트로 핵심 로직 검증" |

### MLOps 관점에서의 의미

MLOps의 CI/CD는 보통 이런 구조입니다:

```
git push → GitHub Actions 트리거
  └→ CI: pytest 실행 → 통과해야만 다음 단계 진행  ← (현재 이 부분이 비어있음)
  └→ CD: main.py 파이프라인 실행
```

현재 [mlops_cron.yml](file:///c:/Users/dlaxo/OneDrive/문서/GitHub/ML-HR/.github/workflows/mlops_cron.yml)은 테스트 없이 바로 `python src/main.py`를 실행하므로, 코드가 깨져도 그대로 실행되는 구조입니다.

왜 좋은 아이디어였는가?
현재 코드처럼 GPT에게 단순히 "0~100점으로 채점해줘"라고 만든 커스텀 프롬프트는, "그 점수를 어떻게 믿어?" 라는 질문에 스스로 증명(인간 평가와 대조)해야 하는 부담이 있습니다.

하지만 QAFactEval과 G-Eval은 이미 세계적인 학회(ACL 등)에서 논문으로 검증된 지표입니다.

"논문 저자들이 실험해보니 이 방식이 인간 평가와 90% 일치하더라"라는 학술적 근거가 이미 존재합니다.
따라서 이 지표들을 파이프라인에 그대로 구현해 냈다면, 굳이 우리 프로젝트 안에서 따로 평가 기준을 검증할 필요 없이 **"학술적으로 증명된 평가 방식을 도입했다"**고 방어할 수 있었습니다.
