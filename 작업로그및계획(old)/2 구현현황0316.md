# ML-HR 구현 현황 정리

기준 업데이트일: `2026-04-07`

## 1. 문서 목적

이 문서는 현재 저장소에 반영된 기초 파이프라인이  
프로젝트 아키텍처 기준으로 어디까지 구현되었는지 정리한 문서입니다.

정리 기준은 다음과 같습니다.

- 아키텍처 단계별 구현 여부 확인
- 현재 구현 범위와 미구현 범위 구분
- 다음 우선 개발 항목 도출

## 2. 검토 대상

- 구현 저장소: `ML-HR`
- 확인한 주요 파일:
  - `src/collect.py`
  - `src/preprocess.py`
  - `src/evaluate.py`
  - `src/auto_pipeline.py`
  - `run_all.bat`
  - `README.md`

## 4. 현재 구현된 범위

### 1) 데이터 수집
- 네이버 뉴스 API를 호출해 뉴스 데이터를 가져오는 기능이 있습니다.
- 기사 링크를 열어 본문을 추출하려는 로직도 포함되어 있습니다.
- 수집 결과는 `Supabase`의 `news_data` 테이블에 저장되도록 작성되어 있습니다.

### 2) 기초 전처리
- HTML 태그 제거
- 이메일 제거
- 특수문자 정리
- 다중 공백 정리

현재는 텍스트 클리닝 수준의 전처리만 구현되어 있습니다.

### 3) 다중 모델 요약 생성
- 아래 3개 모델에 대한 요약 생성 코드가 작성되어 있습니다.
  - `KoBART`
  - `KoT5`
  - `KLUE-RoBERTa`

- 모델별 요약 결과를 DB 컬럼에 저장하는 구조가 들어 있습니다.
- 다만 이것은 “평가 파이프라인”보다는 “다중 모델 추론 데모”에 가깝습니다.

### 4) 단일 모델 자동 요약 스크립트
- `KoBART` 기반의 자동 요약 스크립트가 별도로 존재합니다.
- 이 스크립트는 `summary` 컬럼을 채우는 구조입니다.
- 현재 메인 실행 흐름과는 분리되어 있어 기준 파이프라인으로 보기 어렵습니다.

## 5. 아직 구현되지 않은 핵심 범위

### 1) 모델 평가 및 비교
- `ROUGE`
- `Latency`
- `Cost`
- `Stability`

위 지표를 계산하고 비교하는 로직이 아직 없습니다.  
따라서 현재는 “누가 더 좋은 모델인지” 자동으로 판단할 수 없습니다.

### 2) MLflow 기반 실험 관리
- `mlflow.db`, `mlartifacts` 폴더는 존재합니다.
- 하지만 실제 코드에서 `mlflow`를 사용해 실험 로그를 남기지는 않습니다.

즉, 현재는 실험 관리 체계가 연결된 상태가 아닙니다.

### 3) 최우수 모델 선정 및 Model Registry
- 여러 모델 결과를 저장하는 것까지만 되어 있습니다.
- 최종 우수 모델을 고르고 등록하는 로직은 아직 없습니다.

### 4) 배포 파이프라인
- `FastAPI`
- `Docker`
- `Render`

관련 코드는 아직 없습니다.  
아키텍처의 배포 단계는 시작 전 상태입니다.

### 5) 모니터링 및 재학습
- `Evidently AI`
- `Prometheus`
- `Grafana`
- 재학습 트리거

모두 아직 구현되지 않았습니다.

## 6. 구현 수준 요약

현재 구현 수준을 크게 보면 아래와 같습니다.

- `데이터 수집 / 저장`: 40~50%
- `전처리`: 20~30%
- `다중 모델 추론`: 30~40%
- `평가 / 비교`: 10% 이하
- `배포`: 0%
- `모니터링`: 0%

## 7. 현재 코드의 성격

현재 파이프라인은 아키텍처 전체를 구현한 상태라기보다,  
아래 단계까지 확인한 “기초 데모”에 가깝습니다.

`뉴스 수집 -> 간단 전처리 -> 여러 모델 요약 생성 -> DB 저장`

즉, 프로젝트의 앞단은 일부 구현되었지만,  
핵심 차별점인 `평가`, `선발`, `배포`, `모니터링`은 아직 붙지 않은 상태입니다.

현재 회의 반영 사항:
- `GPT` 계열과 `Mistral-7B`는 제외했습니다.
- 현재 구현 범위는 3개 모델 비교 기준으로 보는 것이 맞습니다.

## 8. 우선순위 제안

다음 작업은 아래 순서로 진행하는 것이 가장 현실적입니다.

### 1순위
- DB 스키마 정리
- 단일 파이프라인 기준 확정
- `summary`와 `summary_*` 컬럼 구조 통일
- 모델 로딩 방식 개선

### 2순위
- `ROUGE`, `Latency` 측정 로직 추가
- `MLflow` 연동
- 모델 비교 테이블 생성

### 3순위
- 최우수 모델 선정 로직 추가
- `FastAPI` 기반 서빙 구현
- `Docker` 적용

### 4순위
- `Evidently AI` 모니터링
- `Prometheus / Grafana`
- 재학습 자동화

## 9. 한 줄 결론

현재 구현은 아키텍처 기준으로  
`수집 -> 간단 전처리 -> 다중 모델 요약 생성 -> DB 저장`까지는 진행되었고,  
`평가 -> 선발 -> 배포 -> 모니터링`은 아직 본격 구현 전 단계입니다.
