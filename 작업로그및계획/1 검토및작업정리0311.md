# ML-HR 검토 및 작업 정리

## 1. 문서 목적

이 문서는 현재까지 작성된 프로젝트 계획서와 아키텍처를 함께 검토한 결과를 정리한 문서입니다.  
팀원들과 상의가 필요한 보완 사항을 한곳에 모으고, 지금까지 진행한 작업도 간단히 기록합니다.

## 2. 프로젝트 계획서 보완 사항

### 평가 기준 구체화
- 모델 선발 기준을 더 명확히 정할 필요가 있습니다.
- 현재는 `ROUGE`, `Latency`, `비용` 중심으로 제시되어 있으므로, 실제 선발 규칙까지 정하면 좋습니다.
- 예시:
  - `ROUGE` 40%
  - `Latency` 30%
  - `비용` 20%
  - `안정성` 10%

### 모델 역할 정리
- `KoBART`, `KoT5`, `GPT-3.5`, `Mistral-7B`는 요약문을 새로 생성하므로 생성형 요약 모델 후보로 자연스럽습니다.
- `KLUE-RoBERTa`는 중요 문장을 선택하는 데 쓰이므로 추출형 기준 모델에 가깝습니다.
- 따라서 모델 후보군을 다음처럼 구분하면 설명이 더 명확해집니다.
  - 생성형 요약 모델
  - 추출형 기준 모델

## 3. 아키텍처 보완 사항

### 저장소 역할 분리
- 현재 아키텍처에서는 `Supabase DB`와 `DVC + S3`의 역할이 다소 겹쳐 보일 수 있습니다.
- 권장 역할 분리는 아래와 같습니다.
  - `Supabase DB`: 수집된 뉴스 원문 및 운영 데이터 저장
  - `DVC + S3`: 학습 데이터셋 버전 관리, 모델 가중치 및 아티팩트 저장

### ETL 또는 전처리 단계 추가
- 현재 데이터 수집 뒤에 바로 DB 저장으로 연결되어 있습니다.
- 실제 구현에서는 `수집 -> 정제/중복 제거 -> 저장` 단계가 있으면 더 현실적입니다.
- 따라서 `Preprocessing` 또는 `ETL` 노드를 추가하는 것이 좋습니다.

### 실험 오케스트레이션 표현 보강
- `SageMaker 실험실`에서 각 모델로 직접 연결되는 구조는 이해 가능하지만, 실험 관리와 모델 후보군이 혼재되어 보일 수 있습니다.
- `Training/Evaluation Pipeline` 또는 `Experiment Pipeline` 단계를 중간에 두면 역할이 더 분명해집니다.

### KLUE-RoBERTa 설명 보강
- 현재는 다른 생성형 모델과 같은 계층에 놓여 있어 역할 차이가 잘 드러나지 않습니다.
- `추출형 기준 모델` 또는 `비교용 Baseline`이라는 설명을 붙이면 질문을 줄일 수 있습니다.

### 배포 승인 단계 고려
- 현재는 `Model Registry -> FastAPI + Docker`로 바로 이어집니다.
- 발표나 설계 문서에서는 `Champion Model` 또는 `Deployment Approval` 단계를 넣으면 운영 흐름이 더 현실적으로 보입니다.

### 모니터링 구조 분리
- 품질 모니터링과 시스템 모니터링을 구분하면 더 완성도 있는 구조가 됩니다.
- 권장 분리는 아래와 같습니다.
  - `Evidently AI`: 데이터 드리프트, 요약 품질 저하 감시
  - `Prometheus / Grafana`: 서버 상태, 응답 속도, 시스템 지표 감시

### 표현 문구 수정
- `1등 모델 출근`은 캐주얼하고 재미있는 표현이지만, 발표 자료나 문서에는 다소 가벼워 보일 수 있습니다.
- `선정 모델 배포` 또는 `최우수 모델 배포`로 바꾸는 편이 더 안정적입니다.

## 4. 지금까지 진행한 작업

### 1) 프로젝트 계획서 검토
- 내용을 읽고 핵심 구조를 확인했습니다.
- 주제 방향과 MLOps 흐름이 잘 잡혀 있다는 점을 정리했습니다.
- 팀원들과 상의가 필요한 보완 사항을 간단히 추려드렸습니다.

### 2) Markdown 계획서 작성
- 기존 텍스트 계획서를 읽기 쉬운 Markdown 형식으로 재작성했습니다.
- 생성 파일:
  - `프로젝트계획.md`

### 3) 아키텍처 검토
- 첨부된 아키텍처 이미지와 Mermaid 코드 파일을 함께 검토했습니다.
- 전체 흐름이 적절한지 확인했고, 저장소 역할 분리, ETL 단계, 모니터링 구조 등 보완점을 정리했습니다.

### 4) 통합 검토 문서 작성
- 계획서 개선점과 아키텍처 개선점을 한 문서로 통합 정리했습니다.
- 현재 문서에서 그 내용을 계속 업데이트하고 있습니다.

### 5) Mermaid 아키텍처 수정본 작성
- 검토 내용을 반영해 Mermaid 코드 수정본을 작성했습니다.
- 개선 항목:
  - 저장소 역할 분리
  - `ETL / Preprocessing` 단계 추가
  - `Experiment Pipeline` 및 `Model Evaluation` 단계 추가
  - `KLUE-RoBERTa` 역할 명시
  - `Deployment Approval` 단계 추가
  - 모니터링 구조 보강

### 6) 좌우 배치용 Mermaid 버전 작성
- 발표 자료에 넣기 쉬운 `Left-to-Right` 레이아웃 버전을 별도로 작성했습니다.
- 데이터 수집부터 모니터링까지 단계가 왼쪽에서 오른쪽으로 읽히도록 구성했습니다.
