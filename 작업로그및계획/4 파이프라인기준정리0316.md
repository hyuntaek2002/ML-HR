이 문서는 2026-04-07 기준으로  
현재 저장소의 공식 실행 흐름과 최근 정리 작업 내용을 간단히 기록한 문서입니다.

## 2. 공식 파이프라인 기준

현재 저장소의 공식 파이프라인은 아래 흐름으로 통일합니다.

`collect.py -> evaluate.py`

- `collect.py`
  - 네이버 뉴스 API 호출
  - 기사 본문 추출
  - `Supabase`의 `news_data` 테이블에 저장

- `evaluate.py`
  - 저장된 기사 조회
  - 3개 모델(`KoBART`, `KoT5`, `KLUE-RoBERTa`) 실행
  - 모델별 요약 결과를 DB에 저장

- `main.py`
  - 공식 실행 엔트리포인트
  - 내부에서 `collect -> evaluate`를 순서대로 실행

## 3. 파일 역할 정리

- `src/main.py`
  - 공식 실행 파일

- `src/collect.py`
  - 뉴스 수집 및 저장 전담

- `src/evaluate.py`
  - 다중 모델 요약 전담

- `src/auto_pipeline.py`
  - 단일 `KoBART` 데모 스크립트
  - 메인 파이프라인에는 포함하지 않음

- `run_all.bat`
  - `python src/main.py`만 실행하도록 정리

## 4. DB 기준 스키마

현재 `news_data` 테이블은 아래 컬럼 기준으로 이해합니다.

- `id`
- `title`
- `description`
- `originallink`
- `topic`
- `summary_kobart`
- `summary_kot5`
- `summary_roberta`

정리 원칙은 다음과 같습니다.

- 모델별 결과 저장 구조를 표준으로 사용
- 기존 `summary` 컬럼은 레거시 컬럼으로 간주
- 메인 파이프라인에서는 `summary` 컬럼을 사용하지 않음
